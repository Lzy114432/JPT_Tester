# 系统架构设计（监控型 PLC + 内部流程控制）

你描述的目标可以归纳为一句话：

- 外部（PLC/IO/MES/产线）是“输入与确认”，内部（你的流程控制器）是“步骤编排与动作执行”，两者通过消息解耦；UI 只负责按钮与显示。

本设计基于 `EwanCommon` 已抽取的能力组合：

- 初始化/生命周期（可选）：`EwanCore.Bootstrap.ManagerLifetimeHost`
- 后台轮询/后台任务：`EwanCore.Runner.StreamRunner`（承载多个采集/推送模块）
- UI/流程：`EwanCore.StateMachine.LogicBase` + `LogicRunner` + `MachineOperator`
- 消息解耦：`EwanCore.Messaging.MessageBus`（`Publish` 同步 + `Post` 异步队列）+ `MessageHub.Current`
- 报警：`EwanCore.AlarmSystem.AlarmService` + `PlcAlarmTracker<T>`
- MES 双向确认：`EwanCore.Messaging.MessageBus` 内置 `RequestAsync/Respond`（CorrelationId）

---

## 1. 组件分层与职责

### 1.1 UI 层（WinForms/WPF）

- 只做：按钮（启动/停止/暂停/复位/清警）、显示（状态/报警/外部数据）、订阅消息刷新界面
- 不做：PLC/MES 网络、业务轮询、流程“步骤推进”

> UI 线程切换建议用：`MessageBusExtensions.SubscribeOnCurrentContext<T>()`（无 WinForms/WPF 依赖）。

### 1.2 流程层（内部步骤编排）

- 以 `LogicBase` 写步骤：`switch (SwitchIndex)`（你说的“我控制的系统步骤”就在这里）
- 依赖接口/服务（DI 注入）：你的“流程控制器/动作执行器”、外部状态只读快照、MES 网关、报警服务等
- 与外部系统交互：通过 `IMessageBus.RequestAsync(...)` 发请求并等待确认结果（状态机常用“发送一步 + 等待一步”实现）

### 1.3 后台服务层（采集/推送/网关）

建议用 `StreamRunner` 承载多个后台模块（每个模块聚焦一个职责）：

- PLC 轮询模块：读取外部系统状态，更新只读快照，并 `Post(new PlcSnapshotUpdated(...))`
- IO 轮询模块：读 IO/传感器，`Post(new IoSnapshotUpdated(...))`
- MES 发送模块：订阅 `MesRequest`，真实发送，`Post(new MesResult(...))` 回流程
- MES 心跳/产线推送模块：订阅业务消息，向外推送
- 可选：数据落库/日志增强模块

### 1.4 基础设施层（通用能力）

- `MessageBus`：
  - `Publish<T>`：进程内同步通知（状态变化/轻量事件）
  - `Post<T>`：异步队列分发（跨线程解耦、限流/背压、请求/响应）
- `AlarmService`：统一报警中心（外部监控报警 + 内部流程报警 + MES 通讯报警都能接入）
- `RequestAsync/Respond`：基于 CorrelationId 的 Request/Reply（总线内部维护等待表）

## 2. 初始化与可插拔（推荐做法）

新框架推荐把“Composition Root”做清楚：

1) 创建并配置消息总线

```csharp
using EwanCore.Messaging;

var bus = new MessageBus(new MessageBusOptions
{
    AsyncQueueCapacity = 1024,
    OverflowStrategy = MessageOverflowStrategy.DropOldest,
});
MessageHub.Current = bus;
```

2) 注册基础组件为单例（DI 容器/手写容器均可）

- `IMessageBus`：`bus`
- `IAlarmService`：`new AlarmService()`
- 不需要手工注册等待表（`RequestAsync` 内部会按 ReplyType 建立等待表；需要自定义时再用 `RequestAwaiter`）
- 其它网关/驱动：PLC/MES/IO/数据库等接口实现

3) 后台模块与流程逻辑只依赖接口（可测试/可替换）

## 3. 消息契约约定（建议）

不再使用 topic 字符串，统一用“消息类型”表达语义。建议约定：

- 命名空间表达域：`MyProject.Messages.Plc/*`、`MyProject.Messages.Mes/*`、`MyProject.Messages.Control/*`
- 类名表达动作/事件：`PlcSnapshotUpdated`、`MesRequest`、`MesResult`、`ControlStopRequested`

示例：

```csharp
using EwanCore.Messaging;
using System;

public sealed class MesRequest : IMessage, ICorrelatedMessage<Guid>
{
    public Guid CorrelationId { get; set; }
    public DateTimeOffset Timestamp { get; set; }
    public string Api { get; set; }
    public object Payload { get; set; }
}

public sealed class MesResult : IMessage, ICorrelatedMessage<Guid>
{
    public Guid CorrelationId { get; set; }
    public DateTimeOffset Timestamp { get; set; }
    public bool Success { get; set; }
    public string Error { get; set; }
    public object Data { get; set; }
}
```

## 4. 监控型 PLC：只做外部状态输入

你的 PLC 是“外部系统监控”，推荐单向流：

1) PLC 轮询读取外部状态 -> 更新 `ExternalSnapshot`（你自己定义的只读快照）
2) 同时调用 `PlcAlarmTracker.Process(plcModel)` 同步报警
3) 必要时 `Post(new PlcSnapshotUpdated(snapshot))` 给 UI/其它模块显示（不是给流程推进用）

注意：

- 外部报警位还在时，`ClearAlarm()` 只是清掉内存报警列表；下一次轮询仍会重新加回来（这是监控型语义）

## 5. MES 双向确认：流程发请求，后台发/收，流程等结果

这是你描述的核心闭环：`LogicBase -> MES 后台 -> LogicBase`

### 5.1 后台 MES 模块：订阅请求并发送，推回结果

- 订阅：`MesRequest`
- 发送：HTTP/TCP/SDK（你实现）
- 推回：`MesResult`
- 失败：`AlarmService.AddAlarm(... needReset:false)`，并返回一个 `MesResult(Success=false)` 让流程决定停机/重试

### 5.2 流程逻辑：发请求 + 等确认

推荐用内置 `RequestAsync<MesRequest, MesResult>`：

- 进入“发送”步骤：`_pendingTask = _bus.RequestAsync<MesRequest, MesResult>(new MesRequest{...}, timeoutMs: ...)`
- 进入“等待”步骤：检查 task 是否完成；完成则继续，否则继续等待/超时报警

MES 通讯失败报警“无需复位”的实现方式：

```csharp
alarms.AddAlarm("MES发送失败", AlarmLevel.H, needReset:false, key:"Mes.SendFail");
```

清掉这个报警后即可重新启动流程（不要求 Home）。

## 6. 内部流程控制器怎么接入（示例）

你现有“动作执行/硬件写入/内部控制器”继续保留，只需要作为依赖注入给 `LogicBase`：

```csharp
using EwanCore.AlarmSystem;
using EwanCore.Messaging;
using EwanCore.StateMachine;
using System;
using System.Threading.Tasks;

sealed class MainLogic : LogicBase
{
    private readonly IMyFlowController _flow;
    private readonly IExternalSnapshot _ext;
    private readonly IAlarmService _alarms;
    private readonly IMessageBus _bus;

    private Task<MesResult> _pendingTask;

    public MainLogic(IMyFlowController flow, IExternalSnapshot ext, IAlarmService alarms, IMessageBus bus)
    {
        _flow = flow;
        _ext = ext;
        _alarms = alarms;
        _bus = bus;
    }

    public override void Handler()
    {
        switch (SwitchIndex)
        {
            case "初始状态":
                SwitchIndex = "等待外部Ready";
                break;

            case "等待外部Ready":
                if (_ext.IsReady) SwitchIndex = "执行内部步骤";
                else if (Tw.StartCheckIsTimeout(SwitchIndex, 5000))
                    _alarms.AddAlarm("外部系统未Ready", AlarmLevel.H, needReset:false, key:"Ext.NotReady");
                break;

            case "执行内部步骤":
                if (_flow.RunOnce()) SwitchIndex = "MES确认";
                break;

            case "MES确认":
                if (_pendingTask == null)
                {
                    _pendingTask = _bus.RequestAsync<MesRequest, MesResult>(
                        new MesRequest { Api = "Confirm", Payload = _flow.BuildMesPayload() },
                        timeoutMs: 10000);
                    SwitchIndex = "等待MES返回";
                }
                break;

            case "等待MES返回":
                if (!_pendingTask.IsCompleted) break;
                if (_pendingTask.IsCanceled || _pendingTask.IsFaulted)
                {
                    _alarms.AddAlarm("等待MES超时/异常", AlarmLevel.H, needReset:false, key:"Mes.Timeout");
                    Complete();
                    break;
                }

                var res = _pendingTask.Result;
                _pendingTask = null;
                if (!res.Success)
                {
                    _alarms.AddAlarm("MES确认失败：" + res.Error, AlarmLevel.H, needReset:false, key:"Mes.ConfirmFail");
                    Complete();
                    break;
                }

                SwitchIndex = "后续步骤";
                break;

            case "后续步骤":
                Complete();
                break;
        }
    }
}
```

这个例子里：

- PLC 只提供 `_ext.IsReady` 这种“外部状态输入”
- 内部动作都由 `_flow` 完成
- 和 MES 的交互通过消息总线 + awaiter 完成，流程不直接做网络请求
- MES 错误报警 `needReset:false`，停机后清警即可重启

---

## 7. 停机策略建议

你提出“MES 错误让 LogicBase 停机”，有两种解耦方式：

1) **流程自停（推荐）**：流程收到 `MesResult.Success=false` 后自行 `Complete()`（或切到“结束状态”）
2) **外部停机命令**：后台发布 `ControlStopRequested` 消息，由“流程控制分发器”订阅后调用 `MachineOperator.Stop()`（更解耦，但需要一个小分发器模块）
